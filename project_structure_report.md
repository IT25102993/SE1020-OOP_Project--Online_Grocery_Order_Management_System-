# FreshMart Project Structure & Naming Conventions Report

This report provides a comprehensive overview of the **FreshMart** project's folder structure, naming conventions, and architectural patterns. It is designed to help new team members understand how the project is organized and where to find specific components.

## 🏗️ Architectural Overview

The project follows a **Layered Architecture** (often referred to as an N-Tier architecture) using the Spring Boot framework. This separation of concerns ensures that the code is maintainable, testable, and scalable.

### 1. Root Directory Structure

- **`.idea/`**: JetBrains IntelliJ IDEA project-specific settings.
- **`src/`**: The source code of the application.
- **`target/`**: Compiled files and build artifacts (generated by Maven).
- **`uploads/`**: Directory for storing user-uploaded files, such as product images or profile pictures.
- **`pom.xml`**: The Maven Configuration file, containing project dependencies, plugins, and build settings.
- **`database-schema.sql`**: SQL scripts for initializing the database structure.

## 💻 Backend Structure (`src/main/java/com/freshmart`)

The backend is organized into functional layers:

### 📂 `config`
Contains configuration classes for the application. This includes security settings, CORS configuration, and third-party service integrations (like Email).
*   *Naming Convention*: `Config` suffix (e.g., `SecurityConfig.java`).

### 📂 `controller`
The **Web Layer**. Controllers handle incoming HTTP requests (GET, POST, etc.), extract data, and call the appropriate service methods. They then return responses to the client.
*   *Naming Convention*: `Controller` suffix (e.g., `OrderController.java`).

### 📂 `service`
The **Business Logic Layer**. This is where the core functionality of the application resides. Services process data, perform calculations, and orchestrate interactions between repositories.
*   *Naming Convention*: `Service` suffix (e.g., `ProductService.java`).

### 📂 `repository`
The **Data Access Layer**. These are interfaces that extend Spring Data JPA's `JpaRepository`. They provide methods for performing CRUD operations on the database without writing boilerplate SQL.
*   *Naming Convention*: `Repository` suffix (e.g., `CustomerRepository.java`).

### 📂 `entity`
The **Domain Layer**. Entities are Plain Old Java Objects (POJOs) that represent database tables. They are annotated with JPA annotations like `@Entity`, `@Id`, and `@Table`.
*   *Naming Convention*: Singular noun (e.g., `Product.java`, `Order.java`).

### 📂 `dto` (Data Transfer Objects)
Objects used to transfer data between the client and the server. DTOs are often simplified versions of entities, excluding sensitive information or combining data from multiple entities for a specific API response.
*   *Naming Convention*: `Dto` suffix (e.g., `CustomerDto.java`).

## 🎨 Frontend Structure (`src/main/resources/static`)

The frontend is a classic web-based interface using HTML, CSS, and JavaScript.

- **`static/`**: Contains all public-facing assets.
    - **`css/`**: Stylesheets for the application.
    - **`js/`**: JavaScript files for client-side logic and API interactions.
    - **`images/`**: Static images used in the UI.
    - **Top-level HTML files**: Individual pages like `index.html`, `login.html`, `profile.html`, etc.

---

## 📝 Naming Conventions Summary

- **CamelCase**: Used for all Java files and packages (e.g., `FreshMartApplication.java`).
- **kebab-case**: Often used for CSS files and some HTML assets.
- **Suffixes**: Mandatory for layered components (`Controller`, `Service`, `Repository`, `Dto`, `Config`).

---

## 🚀 Why this structure?

1.  **Scalability**: Adding new features is easy—just follow the pattern (Entity -> Repository -> Service -> Controller).
2.  **Maintenance**: If a bug is found in the business logic, you know exactly which `service` to look in.
3.  **Team Collaboration**: Everyone knows where code belongs, reducing merge conflicts and confusion.
